LESSON 0005 · DISTRIBUTE

Ship it — the real procedure, to TestFlight

Every lesson so far, assembled into the exact ordered checklist for your current goal: a build in a tester's hands.

This is where L0001–L0004 stop being separate ideas and become one sequence you actually run. Nothing here is new — it's L0002's chain and L0004's archive/export steps, in the order you'll actually click through them.

The checklist

  1. Confirm the Bundle ID matches a registered App ID. One-time setup per app (L0002) — rarely revisited unless you add a capability like Push, which needs an explicit App ID.
  2. Confirm signing is resolved. With "Automatically manage signing" on and the right team selected in Xcode, a Distribution certificate and an App Store provisioning profile should already exist or get created on demand. If not, that's L0002's chain — check Certificates, Identifiers & Profiles.
  3. Bump the build number (CFBundleVersion). Every single upload to App Store Connect needs a build number strictly higher than the last one, even between TestFlight-only builds. The user-facing version string (CFBundleShortVersionString, e.g. "1.2.0") only needs to change for a real release.
  4. Select "Any iOS Device (arm64)" as the run destination (never a Simulator — L0004), then Product → Archive.
  5. Wait for the Organizer to open automatically and confirm the new archive shows the version/build you expect.
  6. Distribute App → App Store Connect → Upload. Xcode re-validates signing for this destination and uploads directly — no separate export-then-upload step needed for this path.
  7. Wait for Apple's processing — typically 5–30 minutes, longer at busy times. The build appears under the app's TestFlight tab in App Store Connect once done. You may be prompted for export compliance information (encryption usage) the first time.[1]
  8. Add the build to a testing group. Internal testers (your own App Store Connect team, up to 100 people) get it within minutes, no review. External testers (up to 10,000, via a link or email) require Apple's Beta App Review on the first build of each version only — typically fast, subsequent builds of the same version are usually near-instant.[1]
  9. Tester installs via the TestFlight app and gets notified of new builds automatically. Builds expire 90 days after upload — after that, testers see "Expired Build" and can't launch it.

the Expo/EAS delta eas build --platform ios followed by eas submit automates steps 2–7 above. EAS (Expo Application Services, Expo's cloud build/submit service) either generates and stores your Distribution certificate + provisioning profile for you ("managed credentials"), or uses ones you supply locally — but it's still creating and using the exact same App ID / certificate / provisioning profile objects on Apple's servers, just through an App Store Connect API key acting on your behalf instead of your own Xcode login.[2] Nothing about L0002's chain changes — only who's operating the client.

Check it sticks

① Walk the checklist from memory, start to finish — you don't need exact wording, just the right steps in the right order.

Confirm App ID matches Bundle ID → confirm signing resolved (cert + profile) → bump build number → select a real-device destination and Archive → Organizer opens, verify the archive → Distribute App → App Store Connect → Upload → wait for Apple's processing → add build to a testing group (internal = instant, external first build = review) → tester installs via TestFlight, build expires in 90 days.

② Which of these requires Apple's Beta App Review?

Internal testers never need review. External testers need review only for the first build of each version — later builds of that same version are typically approved automatically unless something review-relevant (entitlements, privacy strings) changed.
