LESSON 0004 · BUILD

What Product → Archive actually does

The moment where systems 1 and 2 (L0001) meet — code becomes a signed, distributable artifact.

Everything from L0002 (the trust chain) and L0003 (the project file) converges here. Archiving isn't a special magic build — it's an ordinary build with two differences: it always uses Release, and it always gets signed for distribution, not just for your own device.

What happens when you choose Product → Archive

  1. Xcode compiles the active scheme's Release configuration — not Debug. This is why an archive can behave differently from what you saw while debugging: optimizations are on, debug-only code paths are off.
  2. The destination must be a real device or "Any iOS Device (arm64)" — never the Simulator. Simulator builds aren't signed the way a real device or App Store Connect requires, so Xcode won't offer to archive one. This is the single most common "Archive is greyed out" confusion — check your selected destination first.
  3. Code signing resolves: Xcode finds a Distribution certificate and a matching provisioning profile for your Bundle ID (automatically, if signing is managed automatically) and signs the compiled binary with them — this is L0002's chain being walked for real.
  4. Everything is assembled into a .app bundle — binary, resources, embedded frameworks, Info.plist.
  5. .dSYM files are generated — debug symbol files mapping the compiled binary back to source file/line, needed later to turn a crash report's raw memory addresses into a readable stack trace.[1]
  6. All of it is packaged as a .xcarchive, stored under ~/Library/Developer/Xcode/Archives, along with metadata: bundle ID, version, build number, signing identity used.

An .xcarchive is not yet distributable on its own. It's an intermediate artifact — the next step, export, is what turns it into something a device or App Store Connect will actually accept.

Export: a separate, deliberate step

Once archiving finishes, Xcode opens the Organizer window (Window → Organizer) listing your archives. Choosing Distribute App here starts export — re-packaging the same archive for a specific destination, each with its own signing requirements:

MethodRe-signed withResult
App Store ConnectApp Store distribution profileUploaded directly (or exported as a package) for TestFlight / App Store review
Ad HocAd Hoc distribution profileAn .ipa installable only on the profile's registered devices
EnterpriseEnterprise distribution profileAn .ipa for in-house distribution, no App Store involved
DevelopmentDevelopment profileAn .ipa for your own registered devices, outside Xcode's Run button

The archive itself is signed once at archive time — export can re-sign it differently depending on where it's headed, which is why the certificate/profile Xcode used to build isn't necessarily the exact one that ends up in what a tester installs.

reading a failure "No signing certificate found," "profile doesn't match this device," "no matching provisioning profile" — these are always step 3 (code signing resolution) or the export re-sign step failing to find a valid link in L0002's chain. The fix is never in your Swift code; it's in Certificates, Identifiers & Profiles or in Xcode's Signing & Capabilities tab.

Check it sticks

① Which build configuration does an archive always use, and why does that matter if you're chasing a performance bug?

Release, always — never Debug. It matters because Release has optimizations on and debug-only code paths off, so a performance problem (or a bug) that only shows up in Release won't reproduce while just running/debugging normally — you have to archive (or switch your run scheme to Release) to see it.

② Your run destination in Xcode's toolbar is set to "iPhone 16 Pro Simulator." Why is Product → Archive greyed out?

Archiving exists to produce a signed, distributable artifact. Simulator builds target a different architecture and were never signed with a distribution-relevant certificate/profile, so Xcode won't offer to archive one — switch the destination to a real device or "Any iOS Device (arm64)" first.

why the dSYM matters later A crash report from a tester's device only contains raw memory addresses — useless without the exact .dSYM that was generated for that exact build. App Store Connect keeps these paired with the build you uploaded automatically; if you ever export/upload manually or via a third-party tool, losing track of dSYMs is a real, common way to end up with unsymbolicated, unreadable crash logs.
