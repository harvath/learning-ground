LESSON 0002 · SYSTEM 2

Identity & trust — the code-signing chain

What "Automatically manage signing" is quietly doing for you, one link at a time.

L0001 named the shape: your Mac and Apple's servers don't trust each other by default, and a chain of registered objects is what bridges them. This lesson opens that chain link by link — it's the direct answer to "what's an App ID, and why do I have to register it."

The chain, in order

Apple Developer Program membership
A paid enrollment (per Apple Account) that unlocks everything below. Assigns you a Team ID — a 10-character ID every other object in this chain is scoped to.
App ID
Your Bundle ID (com.you.app, set in Xcode) registered on Apple's servers, plus the list of capabilities it's allowed to use. Nothing downstream can exist until this does — a provisioning profile literally references an App ID.[1]
Certificate
Proves a key pair belongs to your Team. You generate the pair — via Keychain Access's Certificate Assistant, or automatically by Xcode — and only the public key ever leaves your Mac, wrapped in a Certificate Signing Request (CSR). Apple's own Certificate Authority signs that public key and hands back the certificate.[4] The private key is never generated by Apple and never sent to it — it stays in your Keychain, and it's what actually performs the signing at archive time. Development certificates sign builds for your own registered devices; Distribution certificates sign builds headed to TestFlight/App Store.
Provisioning profile
Generated by Apple, this is the glue: it bundles one App ID + authorized certificate(s) + granted entitlements + (for Development/Ad Hoc) a device UDID list into one file. This is what Xcode embeds into the build and what a device or App Store Connect checks before allowing it to run.[2]
→ device (Development/Ad Hoc: must be on the profile's device list) or App Store Connect (App Store: no device list — Apple re-signs for delivery instead)

Every one of these has to exist, and has to reference the one above it correctly, before iOS or App Store Connect will accept a build. That's the literal, mechanical answer to "why register at developer.apple.com": a provisioning profile cannot be issued without an App ID to reference, and nothing runs on a real device or reaches TestFlight without a valid provisioning profile.

why two separate proof-files, not one Certificate and provisioning profile answer two different questions that change on different schedules. The certificate answers "who is this?" — an identity, scoped to your Team, unaware of any particular app. The profile answers "is this app, signed by that certificate, allowed to run here, with these capabilities?" — authorization, scoped to one App ID.[3] Splitting them lets each rotate independently: one Distribution certificate signs every app your Team owns and only changes on an identity cadence (yearly expiry, or immediately if a key is compromised); a profile changes constantly and per-app instead — add a test device, add a capability, target a new distribution channel. Merge them into one file and every small per-app tweak would force you to re-touch your actual identity proof, and every identity rotation (new Mac, expired cert) would force regenerating authorization data for every app you own. The relationship is also many-to-many, not one-to-one — a profile can list several authorized certificates (you, a teammate, a CI machine, all allowed to build the same app), and one certificate is referenced by many profiles across your different apps — which a single merged object couldn't express without duplicating identity data everywhere.

Explicit vs. wildcard App IDs

An explicit App ID matches exactly one Bundle ID (com.you.app) and is required for capabilities that need to be individually identified — Push Notifications, In-App Purchase, most modern capabilities. A wildcard App ID (com.you.*) covers many apps at once but can't use those capabilities. If your app uses Push or similar, it needs an explicit App ID — there's no way around registering one per app.

what "automatically manage signing" is actually doing That checkbox in Xcode's Signing & Capabilities tab doesn't skip the chain above — it automates walking it, using your logged-in Apple Account. Behind the scenes Xcode: checks whether an App ID matching your Bundle ID exists (creates one if not), checks whether you have a valid certificate for the current purpose (requests one if not), and checks whether a provisioning profile ties them together correctly for your current registered devices (creates/renews one if not). Turn it off and you're doing each of those steps by hand on developer.apple.com — same chain, manual walk.

Registering a device works the same way. With automatic signing on, Xcode registers a newly-connected device's UDID with your Team the first time you try to run on it. By hand, that's Certificates, Identifiers & Profiles → Devices → add the UDID yourself.[5] Either way, a Development/Ad Hoc profile only ever lists whichever devices were registered at the time it was generated — plug in a new device and the old profile simply won't include it until it's regenerated.

TestFlight — the user's current channel — rides on the App Store profile type. That's why adding testers doesn't mean registering their devices: Apple's own signing takes over once a build reaches App Store Connect.

how tampering is actually prevented Two separate signatures are doing this, not one. The provisioning profile itself is a signed file — Apple wraps it in a CMS/PKCS#7 signature, the same signing format used elsewhere in cryptography, not something iOS-specific.[6] Edit a single byte of it — say, to sneak in a device UDID that was never really registered — and Apple's signature over the file no longer matches its contents, so the OS's check of that signature fails and the whole profile is rejected outright. The app binary is protected separately, by code signing proper: at archive time, every file in the signed bundle gets hashed, those hashes are bundled into a code signature, and that signature is signed with your certificate's private key. At install or launch, the OS re-hashes what's actually on disk right then and checks it against that signature — change even one byte after signing and the hashes stop matching, so it won't run. Both checks ultimately verify against Apple's own root certificate, the one thing baked into the OS as unconditionally trusted — your certificate is only trusted because Apple's own Certificate Authority signed it. It's signatures chained all the way up, terminating at one thing only Apple controls.

Check it sticks

① What four things does a provisioning profile bundle together?

The App ID, the authorized certificate(s), the granted entitlements, and — for Development/Ad Hoc profiles only — the list of allowed device UDIDs.

② In one sentence: why can't a provisioning profile exist before an App ID is registered?

A provisioning profile's whole job is to authorize a specific App ID for a specific certificate/entitlements/device combination — it references the App ID as one of the things it's bundling, so there's nothing for it to reference until the App ID exists.

③ Two different things get tamper-protected here — the provisioning profile, and the app binary. What actually stops each one from being silently modified?

The provisioning profile is itself a file Apple cryptographically signs (CMS/PKCS#7) — any edit to it breaks Apple's signature, which the OS checks before trusting it. The app binary is protected by code signing: every file in the bundle is hashed at archive time, the hashes are signed with your certificate's private key, and the OS re-hashes and re-checks at install/launch — any change after signing breaks the match.

④ Which profile type does not require a registered device UDID list?

App Store profiles skip the device list because Apple's own infrastructure re-signs and delivers the build (via the App Store or TestFlight) — the device-level trust check happens on Apple's side, not baked into your profile.

go look at the real thing Open developer.apple.com → Certificates, Identifiers & Profiles → Identifiers and find your app's App ID. Note exactly which capabilities are checked. Then open Xcode's Signing & Capabilities tab for the same target and compare. They should match exactly — if they don't, that mismatch is a real, current source of signing errors, not a hypothetical one.
