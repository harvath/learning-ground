LESSON 0001 · THE BIG PICTURE

Three systems, two proof-files, one pipeline

Before any term-by-term detail: the shape of the whole thing, so every later lesson has somewhere to attach.

Building and shipping an iOS app feels like a pile of unrelated trivia — App IDs, certificates, archives, xcodeproj — because each one shows up in a different tool at a different moment. They're not unrelated. They're three systems that don't trust each other by default, plus two files whose entire job is to carry proof between them. Once that shape is in your head, every individual term is just "which system, which proof."

The three systems

1. Your Mac — the Xcode project
Your code, plus the .xcodeproj that describes how to build it. This is the only system you fully own. Xcode can compile and run this on the Simulator forever without talking to Apple at all.
2. developer.apple.com — identity & trust registry
Where your Apple Developer Program membership lives. This is where you register App IDs, hold certificates, and issue provisioning profiles. Its whole job is answering one question: "is this app, signed by this key, allowed to run on that device?"
3. App Store Connect — distribution
A separate website from developer.apple.com (historically called iTunes Connect — old tutorials still use that name), but backed by the same membership and Team ID. This is where an exported build becomes a TestFlight beta or an App Store release.

System 1 (your Mac) is enough on its own to build and run the app on the Simulator — systems 2 and 3 aren't involved yet. The moment you want it on a real device — yours, a tester's, or the public's — system 2 has to vouch for the binary before system 1 or 3 can hand it off. That's the whole reason registration exists, and it's the subject of L0002.

The two proof-files

Systems 1 and 2 don't share a login or a filesystem. They exchange trust through two artifacts:

Certificate — proves who signed this. Issued by Apple to your Developer Program membership; the matching private key lives only in your Mac's Keychain.

Provisioning profile — proves this specific app, signed by this specific certificate, is allowed to run here. It bundles the App ID, the authorized certificate(s), granted entitlements, and (for development/testing profiles) the exact devices allowed.

Every build you ever archive or run on a device gets both of these baked in. When something fails with a signing error, it is always one of these two files — missing, expired, or not matching what the build is trying to do.

The pipeline, end to end

  1. Write code in an Xcode project that has a chosen Bundle ID.
  2. Register identity on developer.apple.com: an App ID matching that Bundle ID, a certificate, and a provisioning profile that ties them together (often automated by Xcode — L0002).
  3. Product → Archive: Xcode compiles a Release build, signs it using the certificate + profile from step 2, and packages it as a .xcarchive (L0004).
  4. Export / Distribute: the archive is re-packaged for a specific destination — App Store Connect, Ad Hoc, Enterprise, or Development (L0004).
  5. App Store Connect receives the export, processes it, and makes it available via TestFlight or the App Store (L0005).
  6. Apple's own signing takes over from here — a tester's device trusts the build because it trusts Apple, not because it trusts you directly.
why can't I skip step 2 iOS refuses to launch any binary that isn't signed by a certificate Apple issued and authorized — via a provisioning profile — for that exact app and (for dev/testing builds) that exact device. This isn't paperwork for its own sake: it's the actual mechanism the OS uses to answer "did anyone with a real Apple Developer Program membership vouch for this code?" Skipping registration doesn't mean skipping a form, it means the OS has no way to answer that question, so it says no.
reframe for debugging Almost every confusing error you'll hit during Product → Archive or export is not a bug in your code or in Xcode — it's system 1 (Xcode) discovering that system 2 (developer.apple.com's records) doesn't have what it expects: a missing App ID, an expired certificate, a profile that doesn't list your device. Once you know the error is really "which system, which proof," you stop guessing and start checking Certificates, Identifiers & Profiles directly.

Check it sticks

① Name the three systems involved in shipping an iOS app, and what each one owns.

Your Mac / Xcode project (code + project file, fully yours). developer.apple.com (identity & trust: App IDs, certificates, provisioning profiles). App Store Connect (distribution: TestFlight, App Store, review) — a separate site, same membership underneath.

② What are the two files that carry proof between your Mac and Apple's systems, and what does each one prove?

Certificate — proves who signed the build. Provisioning profile — proves that this app, signed by this certificate, is allowed to run on this device/destination (and bundles the App ID + entitlements + certificate + device list together).

③ Which of these can happen with zero contact with Apple's servers?

The Simulator doesn't check code signing the way a real device does — it's not "a real iOS," so none of the trust chain applies. The moment a real device is involved, even just your own, Apple's identity system (App ID, certificate, provisioning profile) has to be in place.
