LESSON 0002 · SYSTEM 2

Identity & trust — the code-signing chain

What "Automatically manage signing" is quietly doing for you, one link at a time.

L0001 named the shape: your Mac and Apple's servers don't trust each other by default, and a chain of registered objects is what bridges them. This lesson opens that chain link by link — it's the direct answer to "what's an App ID, and why do I have to register it."

The chain, in order

Apple Developer Program membership
A paid enrollment (per Apple Account) that unlocks everything below. Assigns you a Team ID — a 10-character ID every other object in this chain is scoped to.
App ID
Your Bundle ID (com.you.app, set in Xcode) registered on Apple's servers, plus the list of capabilities it's allowed to use. Nothing downstream can exist until this does — a provisioning profile literally references an App ID.[1]
Certificate
Proves a private key — generated on your Mac and never sent to Apple — belongs to your Team. Apple signs the public half and hands back the certificate. Development certificates sign builds for your own registered devices; Distribution certificates sign builds headed to TestFlight/App Store.
Provisioning profile
Generated by Apple, this is the glue: it bundles one App ID + authorized certificate(s) + granted entitlements + (for Development/Ad Hoc) a device UDID list into one file. This is what Xcode embeds into the build and what a device or App Store Connect checks before allowing it to run.[2]
→ device (Development/Ad Hoc: must be on the profile's device list) or App Store Connect (App Store: no device list — Apple re-signs for delivery instead)

Every one of these has to exist, and has to reference the one above it correctly, before iOS or App Store Connect will accept a build. That's the literal, mechanical answer to "why register at developer.apple.com": a provisioning profile cannot be issued without an App ID to reference, and nothing runs on a real device or reaches TestFlight without a valid provisioning profile.

Explicit vs. wildcard App IDs

An explicit App ID matches exactly one Bundle ID (com.you.app) and is required for capabilities that need to be individually identified — Push Notifications, In-App Purchase, most modern capabilities. A wildcard App ID (com.you.*) covers many apps at once but can't use those capabilities. If your app uses Push or similar, it needs an explicit App ID — there's no way around registering one per app.

what "automatically manage signing" is actually doing That checkbox in Xcode's Signing & Capabilities tab doesn't skip the chain above — it automates walking it, using your logged-in Apple Account. Behind the scenes Xcode: checks whether an App ID matching your Bundle ID exists (creates one if not), checks whether you have a valid certificate for the current purpose (requests one if not), and checks whether a provisioning profile ties them together correctly for your current registered devices (creates/renews one if not). Turn it off and you're doing each of those steps by hand on developer.apple.com — same chain, manual walk.

Check it sticks

① What four things does a provisioning profile bundle together?

The App ID, the authorized certificate(s), the granted entitlements, and — for Development/Ad Hoc profiles only — the list of allowed device UDIDs.

② In one sentence: why can't a provisioning profile exist before an App ID is registered?

A provisioning profile's whole job is to authorize a specific App ID for a specific certificate/entitlements/device combination — it references the App ID as one of the things it's bundling, so there's nothing for it to reference until the App ID exists.

③ Which profile type does not require a registered device UDID list?

App Store profiles skip the device list because Apple's own infrastructure re-signs and delivers the build (via the App Store or TestFlight) — the device-level trust check happens on Apple's side, not baked into your profile.

go look at the real thing Open developer.apple.com → Certificates, Identifiers & Profiles → Identifiers and find your app's App ID. Note exactly which capabilities are checked. Then open Xcode's Signing & Capabilities tab for the same target and compare. They should match exactly — if they don't, that mismatch is a real, current source of signing errors, not a hypothetical one.
