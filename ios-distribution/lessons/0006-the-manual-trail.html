LESSON 0006 · SYNTHESIS

The manual trail — every step, once, by hand

L0001–L0005, walked as one real sequence instead of five separate concepts.

L0002's "automatically manage signing" box described what Xcode does invisibly: check for an App ID, request a certificate, generate a provisioning profile. This lesson is that same sequence, done by hand, once, narrated — so every click has a name and a reason attached to it, using one real app as the example throughout: EbiHouse, Bundle ID tokyo.harvath.ebihouse, a push-notification app. Swap in your own name and Bundle ID — every step and every reason stays identical.

Step 1 — Register the App ID

  1. developer.apple.com → Account → Certificates, Identifiers & Profiles
  2. Identifiers → +
  3. Choose App IDs → App → Continue
  4. Description: EbiHouse. Bundle ID: Explicit → tokyo.harvath.ebihouse
  5. Under Capabilities, enable Push Notifications
  6. Register
why This is L0002's chain, link one: nothing downstream — no certificate-to-app binding, no provisioning profile — can exist until an App ID exists to reference.[1] Explicit, not wildcard, because Push Notifications is exactly the kind of capability that needs individual identification — a wildcard App ID can't carry it (L0002). This step also plants the one value every later system keys off of: tokyo.harvath.ebihouse has to match, byte for byte, in Xcode's target settings and in App Store Connect (Step 4) — it's the thread tying all three systems from L0001 together.

Step 2 — Create a Distribution certificate

  1. Certificates → +
  2. Choose Apple Distribution (the modern, unified certificate type — covers App Store, TestFlight, and Ad Hoc in one certificate, replacing the old split "iOS Distribution" type)[2]
  3. Apple asks for a CSR, generated on your Mac: Keychain Access → menu Keychain Access → Certificate Assistant → Request a Certificate from a Certificate Authority. Enter your Apple Account email, save to disk.
  4. Upload the CSR, download the returned .cer, double-click to install it into Keychain
why This is L0002's CSR box, for real: Certificate Assistant just generated an RSA key pair locally, wrapped the public half in a CSR, and self-signed it as proof you hold the private half — which never left your Mac. Uploading it is Apple's CA verifying that proof and signing your public key in return. Picking Apple Distribution rather than Development matters here because a Development certificate can only sign builds for your own registered devices (L0002's table) — it can't produce anything TestFlight or the App Store will accept.

Step 3 — Create a provisioning profile

  1. Profiles → +
  2. Choose App Store Connect (the profile type TestFlight rides on — L0002)
  3. Select the App ID: tokyo.harvath.ebihouse
  4. Select the Distribution certificate from Step 2
  5. Name it (e.g. EbiHouse Distribution) → Generate → Download
  6. Double-click the .mobileprovision file to install it
why Steps 3.3 and 3.4 are you, by hand, assembling the exact bundle L0002 described a provisioning profile as being: one App ID + one authorized certificate (+ entitlements, inherited from the App ID's capabilities). No device list this time — the App Store Connect profile type skips it, because Apple's own infrastructure re-signs and delivers the build once it's uploaded (L0002's table, L0004's export-method table). Double-clicking installs the file where Xcode looks for it at archive time — this is the exact object L0004 described Xcode "finding and embedding" during code signing resolution.

Step 4 — Create the app in App Store Connect

  1. appstoreconnect.apple.com → My Apps → + → New App
  2. Platform: iOS. Bundle ID: tokyo.harvath.ebihouse (picked from a list — it has to already exist as an App ID, from Step 1)
  3. Fill in the app name and a SKU — an arbitrary unique string for App Store Connect's own internal bookkeeping (e.g. ebihouse); it's never shown to users
why This is the first appearance of system 3 (L0001) in this trail — a genuinely separate system from developer.apple.com, with its own record for your app: name, metadata, and eventually the TestFlight tab. The Bundle ID is the join key between the two systems — App Store Connect can only offer you Bundle IDs that already exist as App IDs on developer.apple.com, which is exactly why Step 1 had to happen first. A build you upload later (Step 5) attaches to this app record purely by matching Bundle ID.

Step 5 — Archive and upload from Xcode

  1. Generate the Xcode project: cd ios && xcodegen generate (L0003 — this is the disposable .xcodeproj being regenerated from project.yml)
  2. Open EbiHouse.xcodeproj
  3. Set the scheme to EbiHouse, destination to Any iOS Device (arm64) — never the Simulator (L0004)
  4. Product → Archive
  5. When it finishes, in the Organizer: Distribute App → App Store Connect → Upload
  6. Xcode asks about signing — automatic management resolves to the exact certificate + profile built in Steps 2–3; picking manually and selecting that same profile does the identical thing explicitly
why Every earlier step exists so this one has something to resolve against. L0004's "code signing resolves" step is no longer abstract: Xcode is finding the specific certificate from Step 2 and the specific profile from Step 3, matching them against the Bundle ID from Step 1, and signing with them. The upload lands against the app record from Step 4, purely because the Bundle IDs match. After Apple's processing — typically 5–30 minutes, occasionally up to an hour — the build appears under that app's TestFlight tab (L0005).

This is exactly why "automatically manage signing" feels like magic the first time and invisible ever after: Steps 1–3 are mostly one-time setup with a long, slow-changing lifecycle (L0002's "why two proof-files" box), Step 4 is one-time per app, and Step 5 is the only one you'll actually repeat routinely.

Check it sticks

① Split the five steps across L0001's three systems. Which steps belong to which system?

Steps 1–3 (App ID, certificate, provisioning profile) are all developer.apple.com — system 2. Step 4 (creating the app record) is App Store Connect — system 3. Step 5 (Xcode project generation, archive, upload) starts on your Mac — system 1 — and ends by handing off to system 3.

② Why did Step 1 require an explicit App ID rather than a wildcard one?

Wildcard App IDs work fine for apps that need no individually-identified capabilities — plenty of simple apps use them. The moment a capability like Push Notifications, In-App Purchase, or similar is involved, an explicit App ID is required (L0002).
