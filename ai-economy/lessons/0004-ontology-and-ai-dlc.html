LESSON 0004 · ONTOLOGY & AI-DLC, DEEP DIVE

Two contested terms, taken apart

L0002 introduced both lightly. This time: the actual mechanics of each — and the real, credible critiques.

Palantir's Ontology, piece by piece

Palantir's own docs split the Ontology into two families: semantic elements, which describe structure, and kinetic elements, which enable change.[1]

ElementFamilyWhat it is
Object TypeSemanticAn entity or event in the organization — e.g. "Vehicle," "Employee," "Shipment."
PropertySemanticAn Object Type's characteristics/attributes.
Link TypeSemanticA relationship between two Object Types.
Action TypeKineticThe schema for a bounded, permissioned set of edits (plus side effects) that a human or an agent can submit.
FunctionKineticServer-side code, run inside Foundry's governed execution environment, that reads properties, traverses links, and makes edits.
how an agent actually *acts*, not just queries The governance boundary sits at the Action Type level. An agent given access to an Ontology isn't handed open-ended write access to the underlying data — it's scoped to a specific, pre-defined, auditable set of Action Types. "This agent may submit Action X on Object Type Y" is the actual permission being granted, not "this agent may edit anything." That's the mechanism behind Palantir's claim of governed agent autonomy, not a vague promise.

Ontology MCP is the bridge to the rest of the ecosystem from L0003: a Developer Console feature that exposes an application's Ontology resources as MCP tools, so external agents (Claude, Copilot Studio, Gemini Enterprise) can act as MCP clients against it. It exposes exactly three tool classes — read objects, execute pre-approved actions, query data — and the scope is configured per-application by an admin, so an external agent only ever sees the slice an admin explicitly opened up.[2]

The skeptic's case

There's a real, pointed critique here, not just marketing pushback. A February 2026 piece, "Palantir's 'Ontology' Hustle," makes the sharpest version of it plainly: Object Type = table, Property = column, Link Type = foreign key, Action = stored procedure — i.e., this is entity-relationship modeling, a technique Peter Chen formalized in 1976, wearing a philosophy-department word. The argument is that the term's main function is to make people unfamiliar with databases believe they're looking at something new.[3] This isn't a fringe take — a follow-up public debate on the claim reportedly landed 75% of the audience on the critic's side, real traction among data-infrastructure practitioners.[4]

The fair counterpoint: the genuinely hard, valuable part may not be the vocabulary at all, but the governance/permissioning layer around Actions, plus the hands-on work of actually curating an ontology for one organization's messy real-world data (Palantir's "forward-deployed engineer" practice). Whether "ontology" the word is doing real technical work, or just branding on top of familiar database primitives, is a separate question from whether the product is useful.

teaching takeaway Treat "Ontology" as Palantir's product name for a governed semantic layer + write-permission layer, built on conventional data-modeling primitives (entity-relationship modeling + scoped write permissions + an MCP-exposed API surface). Useful to know as a concrete pattern. But recognize both the mechanics and the critique when the term comes up — using it uncritically as if it were self-evidently novel is a small tell.

What a "Bolt" actually is

AI-DLC (AWS's "AI-Driven Development Lifecycle," from L0002) runs on a unit called a Bolt — the smallest iteration across all three phases (Inception, Construction, Operations), sized in hours or days rather than a fixed two-week sprint. AI proposes an execution path scaled to the task's actual complexity, and work is sequenced as Bolt 1, Bolt 2, ... Bolt n within Construction.[5]

Mob Elaboration and Mob Construction — borrowed from mob programming

Both ceremonies explicitly borrow from Woody Zuill's mob programming practice: the whole team working on the same task, at the same time, in the same space.[6]

Mob Elaboration (Inception phase): AI drafts requirements and stories by asking clarifying questions and proposing decompositions; the whole team validates live — compressing what traditionally takes weeks into roughly 2–3 hours. Mob Construction (Construction phase): using the validated Inception output, AI proposes architecture, domain models, code, and tests; the same cross-functional group clarifies technical decisions as they arise and reviews the generated artifacts before acceptance.[5]

Where the human stays in the loop

The stated core philosophy is "AI proposes, human approves." Gates sit at every stage: a human gives initial task clarification, a human approves the AI-refined plan before execution, a human verifies outcomes after execution, and all interactions are logged for audit. AWS frames these as non-optional, not best-practice suggestions.[5]

read the adoption numbers skeptically The specific case studies you'll see cited — Wipro compressing ~3 months of planned work into ~20 hours across five 4-hour mob sessions, a fintech (Dhan/Dun) building an app in 48 hours and shipping the following week, "10–15x productivity" — all trace back to AWS's own re:Invent 2025 session and blog posts, based on "100+ customer experiments" AWS itself ran or co-ran through its "Unicorn Gym" co-engineering program. No independently audited case study or published failure-case data exists yet. Treat these exactly like any vendor benchmark: real, but not neutral.

Is this AWS-specific, or an industry-wide shift with an AWS name?

Worth comparing rather than taking AI-DLC's framing at face value. Microsoft's answer to the same underlying problem is "Agentic DevOps" (introduced at Build 2025) — AI agents embedded across the SDLC via existing tools (Copilot, Azure DevOps AI) — with no fixed ceremony structure like Bolts or Mob Elaboration. Google's public push in this period is reported as ecosystem/investment-focused (a widely cited ~$750M ecosystem investment figure) rather than a competing named methodology.

What's genuinely AWS-specific is the packaging: a named lifecycle with specific borrowed ceremony vocabulary (Bolt, Mob Elaboration, Mob Construction), formalized into phases and human-approval gates, backed by AWS's own reference tooling (Kiro, Amazon Q Developer, Strands Agents SDK, AgentCore). Whether that packaging is a genuine methodological contribution beyond naming rights, versus AWS's brand on a shift every major vendor is describing some version of, doesn't have a settled answer in independent commentary — worth holding as an open question rather than resolving it either way.

Check it sticks

① In Palantir's Ontology, what's the difference between the semantic elements and the kinetic elements?

Semantic elements describe structure: Object Types (entities/events), Properties (attributes), Link Types (relationships). Kinetic elements enable change: Action Types (schema for a bounded, permissioned set of edits an agent or human can submit) and Functions (server-side code that reads/traverses/edits via the governed execution environment).

② What's the skeptic's one-line mapping of Ontology terms onto plain old database terms — and what's the argument behind it?

Object Type = table, Property = column, Link Type = foreign key, Action = stored procedure. The argument: this is 1976-era entity-relationship modeling wearing new, philosophy-flavored branding — the word's main effect is making it look novel to people unfamiliar with databases.

③ A "Bolt" in AI-DLC is best compared to which part of a traditional Scrum team's rhythm?

A Bolt is the smallest iteration unit across all three AI-DLC phases, playing the same structural role a sprint plays in Scrum, but sized to hours or days and scaled to the task's actual complexity rather than fixed at two weeks.

④ Which of AI-DLC's headline adoption claims (Wipro, Dhan/Dun, "10-15x productivity") has been independently audited outside of AWS's own reporting?

Every specific number and case study found traces back to AWS's own re:Invent 2025 session and blog posts, based on AWS-run or AWS-co-run customer experiments. This is the exact "read a funding/claim headline critically" skill from the mission — applied here to a productivity claim instead of a valuation.
